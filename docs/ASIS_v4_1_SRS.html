1. Purpose and Scope

ASIS is an enterprise strategic intelligence system that accepts a strategic question, contextual metadata, and optional user profile memory; orchestrates an 8-agent analysis workflow; validates the resulting strategic brief against a structured schema and quality gate; and presents the result through an interactive web application and PDF export flow.

This SRS covers the current production-oriented implementation state, including the live LLM path, no-silent-fallback behavior, dashboard and analysis interfaces, event streaming, report generation, authentication, persistence model, and deployment path.

Enterprise note. This version is materially hardened for controlled enterprise deployment validation. Full enterprise certification still depends on environment-level controls such as HTTPS/TLS, OAuth provider setup, secret rotation, and live post-deploy verification.

2. Product Overview

2.1 Product goals

  • Produce decision-first strategic briefs for board and executive audiences.
  • Run a multi-agent analytical workflow rather than a single-model monologue.
  • Make framework usage, provenance, confidence, and quality visible to the user.
  • Support enterprise user journeys from onboarding to analysis completion to PDF reporting.
  • Operate in live LLM mode in production without silently degrading to static fallback outputs.

2.2 Product boundaries

  • The active application surface is the asis/ stack, not the legacy root TypeScript prototype.
  • PDF export is supported through the frontend PDF render route and Puppeteer-based generation.
  • Social OAuth is implemented in code but requires provider credentials and callback configuration to be fully live.
  • Optional external enrichment keys are supported but not mandatory for the core multi-agent analysis path.

2.3 Technology baseline

Layer Current implementation
Frontend Next.js 14 App Router, React 18, TypeScript, Tailwind-style utility styling, charting and React Flow visualisation.
Backend FastAPI, Pydantic v2, SQLAlchemy, LangGraph-inspired state orchestration, Celery-compatible task dispatch patterns.
Persistence SQLite on the GCP free-tier path; SQLAlchemy models support promotion to managed relational storage.
LLM routing LiteLLM proxy when configured, otherwise direct Groq OpenAI-compatible API routing for live JSON generation.
Report export Frontend PDF route with Puppeteer render path and backend report orchestration.

3. System Context and Architecture

3.1 Logical architecture

The system is composed of a browser-based presentation layer, a Next.js application layer for UI and same-origin proxying, a FastAPI backend for analysis orchestration and persistence, a relational database for durable records, and one or more LLM providers accessed via LiteLLM or direct Groq API routing.

3.2 Agent orchestration model

Step Agent Responsibility
1OrchestratorNormalizes the board question, decomposes context, and routes specialist work.
2Market IntelMarket sizing, trend synthesis, demand signals, and external structure.
3Risk AssessmentDownside analysis, execution and enterprise risk framing, execution realism.
4Competitor AnalysisCompetitive positioning, rivals, comparable moves, pressure mapping.
5Geo IntelGeopolitical, regulatory, trade, and CAGE-distance considerations.
6Financial ReasoningScenario modeling, commercial logic, bottom-up economics, investment framing.
7Strategic OptionsOption design, recommendation logic, pathway comparison, option scoring.
8SynthesisValidates and enriches the strategic brief scaffold into a final board-ready response.

3.3 LLM routing requirements

  • Production mode shall prioritize live provider responses for all agents.
  • Production mode shall set ALLOW_LLM_FALLBACK=false to prevent silent template-mode degradation.
  • When LiteLLM proxy credentials are absent and GROQ_API_KEY is present, the system shall route directly to Groq.
  • Synthesis shall use a compact patch prompt and a second smaller repair prompt before raising a live-provider failure.
  • Any production live-provider failure shall surface explicitly rather than being disguised as a completed analysis.

3.4 Validated brief contract

The final output must validate against StrategicBriefV4, including required framework outputs, executive summary shape, decision statement prefix rules, implementation roadmap, quality report, citations, and confidence normalization.

4. User Classes and Primary Journeys

User class Description Primary journeys
Analyst / Associate Primary analysis initiator and report consumer. Login, create analysis, monitor progress, review frameworks, export PDF, revisit prior reports.
Manager / Principal Review and challenge user of strategic outputs. Inspect decision banner, confidence, red-team content, pathway options, and enterprise PDF.
Admin / Platform operator Maintains secrets, deployment, and provider configuration. Set environment variables, deploy to GCP, inspect provider health, verify fallback state, rotate secrets.

4.1 Main user flows

  1. User authenticates with email/password or configured OAuth provider.
  2. User opens dashboard and starts a new strategic analysis.
  3. User submits strategic question and structured metadata.
  4. Backend creates an analysis record, streams status and progress events, and executes the 8-agent pipeline.
  5. User monitors live progress through the analysis running screen and agent collaboration view.
  6. On completion, user reviews the strategic brief, confidence, quality, rigour panel, frameworks, provenance, and roadmap.
  7. User exports or downloads the report PDF and accesses archived reports through the reports page.

5. Functional Requirements

5.1 Authentication and session management

FR-002The system shall support Google and GitHub OAuth login when provider credentials are configured.
FR-003The frontend shall expose provider availability truthfully rather than showing unavailable providers as usable.
FR-004All authenticated analysis and report endpoints shall enforce user ownership checks.

5.2 Analysis creation and orchestration

ID Requirement
FR-005The system shall capture the strategic question, organization, industry, geography, and decision type as analysis context.
FR-006The backend shall create an analysis record before pipeline execution and persist lifecycle timestamps and status.
FR-007The pipeline shall execute all 8 agents in sequence and persist agent logs and event traces.
FR-008The synthesis stage shall validate the final brief shape before marking an analysis completed.
FR-009The backend shall sanitize non-JSON-safe payloads before persistence to avoid analysis stalls at completion time.
FR-010The system shall expose analysis events for live progress rendering via the analysis events endpoint.

5.3 Live LLM behavior

ID Requirement
FR-011The system shall use live model outputs in production when a valid provider configuration is present.
FR-012The system shall fail loudly if production live-provider output is unavailable and silent fallback is disabled.
FR-013The system shall mark analyses and agent logs with used_fallback when fallback is intentionally used.
FR-014The synthesis agent shall use compact live prompts and a smaller repair retry path before surfacing a provider failure.

5.4 Strategic brief presentation

ID Requirement
FR-015The analysis detail view shall display decision banner, confidence, executive summary, framework views, roadmap, and collaboration graph.
FR-016The analysis detail view shall surface fallback warnings when any part of the analysis used fallback mode.
FR-017The UI shall remain stable for reading once an analysis is completed, without repeated auto-scroll, graph churn, or destructive refresh behavior.
FR-018The reports page shall provide report archive, confidence, status, and download access for completed analyses.
FR-019The PDF export route shall generate a board-ready report from the validated strategic brief and report metadata.

5.5 Quality and decision support

ID Requirement
FR-020The system shall produce a quality report with grade, checks, and normalized scores for MECE, consistency, context specificity, financial grounding, and execution specificity.
FR-021The system shall normalize decision and overall confidence values to the unit interval before persistence and rendering.
FR-022The system shall maintain framework completeness across the required eight strategic frameworks in the final brief.
FR-023The system shall persist citations, exhibit metadata, and collaboration events to support provenance and explainability.

6.2 Strategic brief schema requirements

  • The final brief shall include eight framework outputs keyed by their canonical framework names.
  • The decision statement shall begin with PROCEED, CONDITIONAL PROCEED, or DO NOT PROCEED.
  • The brief shall include executive summary, quality report, implementation roadmap, exhibit registry, citations, balanced scorecard, and contextual analysis sections.
  • The brief shall support enriched financial, market, risk, and strategic pathway structures introduced in the current hardened version.

9. Deployment and Operations Requirements

9.1 Environment requirements

  • GROQ_API_KEY must be present or a working LiteLLM proxy must be configured for live production analysis.
  • ASIS_DEMO_MODE must be false in production.
  • ALLOW_LLM_FALLBACK must be false in production to prevent silent template-mode responses.
  • NEXT_PUBLIC_API_URL shall point to the frontend same-origin surface for browser traffic on the GCP single-VM path.
  • OAuth requires Google/GitHub client IDs, secrets, and callback URLs before it can be considered live-ready.

9.2 GCP free-tier deployment assumptions

  • The current deployment path uses a single VM and a free-tier oriented compose file.
  • SQLite is used as the durable store on that VM.
  • Inline analysis execution and lightweight background semantics are preferred over a full distributed worker stack.
  • Redis-dependent paths shall degrade safely when Redis is not provisioned on the free-tier VM.

9.3 Operational quality checks

  • Post-deploy health check shall validate backend /v1/health and frontend /api/health.
  • Post-deploy smoke test shall validate login, dashboard, new analysis creation, analysis completion, reports listing, and PDF generation.
  • Provider health verification shall confirm analyses complete without fallback banner when live LLM credentials are configured.

10. Verification and Acceptance Criteria

Area Acceptance criteria
Backend integrity Backend tests pass, including agent wiring, fallback semantics, synthesis compact retry path, and analysis lifecycle/report generation.
Frontend integrity Frontend build passes and type-check passes for the active Next.js application.
Live analysis An analysis can be created, all eight agents run, synthesis completes, and the user is taken to a finished strategic brief.
No silent fallback Production analysis shall either use live providers or surface a provider failure explicitly; it shall not silently return template results.
Reporting Completed analyses appear on reports page and PDF export returns a valid document.
Auth Email/password auth must be operational; OAuth must either be fully functional or truthfully unavailable until configured.
Current validation baseline. The current implementation has been locally validated through backend test execution, frontend production build, and frontend type-check for the active asis/ stack. Live deployment verification remains the gating step for full production sign-off.

11. Open Risks and Enterprise Readiness Notes

  • Open HTTPS/TLS is still an infrastructure requirement for full enterprise rollout.
  • Open OAuth requires live provider console setup and callback verification to be production-complete.
  • Current state Live multi-agent orchestration is now designed to fail loudly instead of silently falling back when provider configuration is broken.
  • Current state Synthesis has been hardened with compact prompt retry behavior because it is the heaviest stage in the pipeline.
  • Current state Dashboard, analysis, and reports flows have been hardened against malformed legacy data and same-origin API issues.

11.1 Recommended next controls before enterprise sign-off

  1. Complete live post-deploy smoke testing on the current GCP VM after the latest backend/frontend commits are deployed.
  2. Rotate exposed provider keys and keep active secrets only in GitHub Actions secrets or runtime environment configuration.
  3. Add TLS termination and production domain routing.
  4. Complete live OAuth testing for Google and GitHub if enterprise SSO is required for launch.
